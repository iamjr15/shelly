# Architecture

The current milestone implements the local host loop:

- `fieldworkd` listens on a Unix socket at `$XDG_RUNTIME_DIR/fieldwork/control.sock` or a per-user temp fallback.
- The socket parent is created as `0700`, must be owned by the current user, and symlink parents are rejected.
- The socket file is chmodded to `0600`.
- `fieldwork` connects over length-prefixed bincode IPC.
- `fieldworkd` also starts an iroh endpoint for remote clients. It accepts length-prefixed MessagePack frames on a single bidirectional iroh stream.
- The daemon's long-lived iroh secret key is stored in the OS keychain. `FIELDWORK_IROH_SECRET_KEY_B64` is a deterministic CI/smoke-test override for non-interactive environments that cannot open the keychain; production runs should leave it unset. `FIELDWORK_IROH_RELAY_URL` can point at a development or Fieldwork-hosted relay; otherwise the daemon uses the configured iroh default relay map. The Fieldwork relay deploy scaffold exists, while hosted relay DNS, Oracle hosts, and credentials remain release gates.
- The workspace enables iroh's `platform-verifier` feature, Fieldwork-owned Reqwest clients use the workspace `rustls` OS-trust path, and relay OTLP uses OpenTelemetry's `reqwest-rustls` native-root feature. `iroh 1.0.0-rc.0` and the relay ACME stack still include `webpki-roots` internally, so the release gate audits Fieldwork-owned TLS construction rather than relying on absence of that transitive package.
- The daemon spawns arbitrary commands inside PTYs with `TERM=xterm-256color` and `COLORTERM=truecolor`.
- Each session keeps a 256 KB PTY byte ring and monotonic byte sequence numbers.
- Each live session also maintains a `wezterm-term` terminal model. Warm reconnect replays bytes from the ring; cold or stale attach renders a synthetic ANSI snapshot from the parsed terminal state. The daemon returns the snapshot with the ring `seq` captured under the same ring/terminal lock window, so the next warm replay starts after the bytes represented by the synthetic snapshot rather than duplicating a repaint that arrived during attach.
- Mobile-core tracks `last_seen_seq` from `Attached.seq` and `Output.seq`, both defined as offsets immediately after the bytes carried in that message. Native services cache that offset per session and call `attach_session_from(id, last_seen_seq)` to request warm replay. A daemon-side lag event is terminal for the current output attachment after one `Lag` frame, and both iOS and Android terminal controllers reattach from the latest tracked offset.
- Mobile dashboards use a long-lived `SubscribeSessions` stream that receives complete replacement session lists after local desktop creation/removal and session state changes. Multiple attach clients can subscribe to the same session. Input from any client writes to the PTY.
- Attached client viewports are tracked per connection. The daemon applies the minimum attached rows/columns to the PTY and debounces resize storms at 100 ms; `pnpm check:daemon-resize` keeps the attach/update/detach scheduling invariant wired into CI.
- Session summaries and scrollback are persisted in encrypted `sessions.redb`.
- Paired device records are persisted separately in encrypted `devices.redb` under hashed row keys, so raw device node IDs and push tokens live only inside encrypted row payloads.
- The persisted payload is encrypted with XChaCha20-Poly1305 by default. The encryption key is stored in the OS keychain via `keyring`; if keychain access is unavailable, the daemon does not write plaintext unless the user explicitly opts out with `fieldwork settings scrollback-encryption off` and restarts `fieldworkd`.
- The local persistence parent directory is forced to `0700`, the `redb` database file is forced to `0600`, and symlinked persistence directories or database files are rejected before opening either store.
- After daemon restart, dead sessions are restored as last-known sessions. Attaching to one replays encrypted scrollback and then emits `SessionExited`.
- Daemon logs are written to daily `daemon.log*` files under `~/Library/Logs/app.fieldwork` on macOS or `$XDG_STATE_HOME/fieldwork` / `~/.local/state/fieldwork` on Linux. At logging startup, the daemon prunes only `daemon.log*` files older than seven days by modification time.
- Daemon and relay logging both run through a tracing sanitizer layer that drops events marked `privacy.level = "user_content"` before downstream logging layers receive them.
- Daemon and mobile crash reporting are not bundled in v1. `fieldwork settings telemetry on|off` records local diagnostics consent only, and daemon OTLP/Honeycomb export remains intentionally absent.
- Agent-aware state inference is limited to detected Claude Code and Codex commands. Claude prompt detection keys off explicit approval/permission phrases rather than every question mark; Codex accepts `type`, `event`, and `status` structured event shapes from single JSON objects or JSONL event streams. Unknown commands only use byte-rate `Idle`/`Working` state.
- The terminal model supplies sanitized last visible line previews for state inference and writes PTY device-status responses back to the child process.
- `FIELDWORK_SESSION_ID` is injected into spawned PTY environments. Local agent hooks can call `fieldwork hook claude-stop` or pipe Codex JSON/JSONL into `fieldwork hook codex-event` to update the owning session over the Unix socket.
- If `FIELDWORK_RELAY_CONTROL_URL` is set, the daemon enables relay-mediated push. It stores a long-lived Ed25519 relay-signing key in the OS keychain, registers its public key with the relay, signs push-token registration/unregistration, and sends hashed `AwaitingInput` events when Claude/Codex inference enters `AwaitingInput`. Transport failures and temporary relay failures retry with `backon` exponential backoff for a bounded 60-second budget; signed retries regenerate nonce and timestamp.
- `fieldwork pair` generates a 5-character Crockford pairing code, builds a compact `PairingTicket { code, node_id, relay_url, addrs, expires_at }`, prints the encoded `fw1…` ticket as a QR plus the bare code for manual entry, and holds the desktop approval prompt open. A device pairs by either scanning the QR (which carries reachability and the code) or typing the code (resolved to the same ticket through the relay rendezvous when `FIELDWORK_RELAY_CONTROL_URL` is configured); both converge on `PairWithCode` over iroh and explicit desktop approval. The daemon owns code generation (`OsRng`), keeps a single active code with a 10-minute TTL, and invalidates that code after 5 wrong in-band attempts. When `FIELDWORK_RELAY_CONTROL_URL` is set the daemon also best-effort publishes `code → ticket_blob` to the relay; when it is unset the daemon logs a warning and skips publish, so the QR path keeps working and only the typed-code path is unavailable. `CONTRACT_VERSION` is `2`.
- Paired iroh clients may list sessions, attach, send input, resize, detach, and register a push token. They cannot create sessions, kill sessions, administer pairing codes, list/remove devices, or emit agent hook events.
- `fieldwork devices remove <name>` removes the stored device record; that node id receives `Unauthorized` on its next iroh connection.
- `fieldwork devices remove <name>` also enqueues relay token unregistration when the removed device had a push token.
- `fieldwork-mobile-core` exposes the iroh client path through UniFFI records, objects, and callback interfaces: `FieldworkClient`, `AttachedSession`, `SessionListSink`, and `ByteStreamSink`.
- The mobile-core API intentionally has no create-session or kill-session method.
- `apps/ios` is a SwiftUI app target that links the generated UniFFI Swift binding and `FieldworkCore.xcframework`, stores the paired daemon record in the data-protection iOS Keychain as this-device-only and available only while unlocked, gates resume and stale input through biometric-only LocalAuthentication, and renders attached PTY bytes with SwiftTerm. Raw terminal `Data` arrivals publish an output revision before optional text fallback decoding, so native rendering is not dependent on UTF-8 conversion.
- The iOS app can pair, list/subscribe, attach, send input, resize, detach, and submit APNs tokens through mobile-core. It does not expose session creation, session killing, or command selection.
- iOS requests APNs permission only after a saved or newly approved pairing exists and biometric unlock succeeds. APNs token callbacks are retained until pairing is available, foreground notifications use fixed generic copy, and notification taps carry only `session_id_hash`; the app resolves the hash against locally fetched sessions after biometric unlock.
- iOS diagnostics sharing is opt-in through Settings or the delayed one-time post-value consent prompt. `MobileTelemetry` persists consent only; no crash-reporting SDK is initialized.
- `apps/android` is a Jetpack Compose app target that links the generated UniFFI Kotlin binding and Android `.so` outputs from `cargo-ndk`, stores the paired daemon record in Android Keystore-backed `EncryptedSharedPreferences`, excludes pairing and queued FCM-token preference files from backup/transfer, gates resume and stale input through biometric-only BiometricPrompt, renders attached PTY bytes with `connectbot/termlib`, and reattaches from cached `lastSeenSeq` offsets after daemon `Lag` or attached-stream errors.
- The Android app can pair, list/subscribe, attach, send input, resize, detach, and submit FCM tokens through mobile-core. It does not expose session creation, session killing, or command selection.
- Android keeps Firebase Messaging auto-init disabled until pairing, requests `POST_NOTIFICATIONS` and syncs FCM tokens only after a saved or newly approved pairing exists and biometric unlock succeeds, renders foreground FCM messages with fixed generic copy, and routes notification taps through a `session_id_hash`-only intent extra resolved locally after biometric unlock. FCM token refresh callbacks only queue trimmed pending tokens in app-private preferences; the paired-and-unlocked ViewModel sync path sends queued/current tokens through mobile-core and clears a queued token only after successful daemon registration.
- Android diagnostics sharing is opt-in through Settings or the delayed one-time post-value consent prompt. `MobileTelemetry` persists consent only; no crash-reporting SDK is initialized, and the telemetry privacy gate inspects existing APK/AAB outputs for removed crash SDK markers.
- `fieldwork-relay` exposes an Axum control plane (in `crates/relay/src/lib.rs`) with `/healthz`, `/v1/version`, `/v1/pair`, `/v1/push/register-token`, `/v1/push/unregister-token`, `/v1/push`, `/v1/pair/publish`, and `/v1/pair/resolve/{code}`. Production deployment serves this listener over Rustls using relay-only `control-plane.crt`/`control-plane.key` systemd credentials; local development can run loopback HTTP when TLS is not required.
- The pairing rendezvous is the relay's optional support for the typed-code path. `POST /v1/pair/publish` is daemon-signed (`x-fieldwork-signature` over `(method, path, body, nonce, ts_ms)`); it stores `code → ticket_blob` (the opaque encoded `fw1…` ticket) with the daemon-supplied expiry, rejects already-expired publishes, and supersedes any prior code for the same daemon (one active code per daemon). `GET /v1/pair/resolve/{code}` is unauthenticated because the short code is the credential, but it is throttled (20 attempts/min per client), returns a uniform `404` whether the format or the lookup failed, consumes the code on a correct guess, and locks a code out after 5 wrong resolves. Codes are persisted in SQLite alongside the other relay state. The blob is opaque reachability only; no terminal content traverses the relay.
- `fieldwork-relay` exposes aggregate Prometheus metrics from a separate metrics app/listener, defaulting to `127.0.0.1:9090`; the listener can be disabled locally with `FIELDWORK_RELAY_METRICS_ADDR=off`.
- `fieldwork-relay` can export OTLP HTTP/protobuf traces from the relay host. Honeycomb is configured with `FIELDWORK_RELAY_OTLP_ENDPOINT`, `FIELDWORK_RELAY_OTLP_SAMPLE_RATE` (default `0.01`), optional `FIELDWORK_RELAY_HONEYCOMB_DATASET`, and a relay-only `honeycomb-api-key` systemd credential or `FIELDWORK_RELAY_HONEYCOMB_API_KEY_PATH`. The relay records only static endpoint names, service metadata, and fixed enum fields such as push platform and event type.
- The relay persists daemon public keys, push-token ownership, and recent replay nonces in SQLite at `FIELDWORK_RELAY_DB_PATH` (default `/var/lib/fieldwork/relay.db`). Push-token ownership rows carry a last-used timestamp, are refreshed on accepted push dispatch, and are pruned after 90 days with no use. The DB directory is chmodded `0700`, and the main DB plus SQLite `-wal`/`-shm` sidecars are chmodded `0600` on Unix.
- Per-daemon push rate counters use a bounded `moka` TTL cache. When relay-only APNs credentials are present, APNs delivery uses an ES256 provider JWT cached for 50 minutes, a persistent reqwest HTTP/2 client with 60-second keepalive pings, lazy first-dispatch connection establishment, and provider connection reuse across dispatches. The APNs payload contains only fixed alert copy plus opaque session hashes. APNs `BadDeviceToken` responses remove the relay token binding from memory and SQLite before the relay reports a fixed stale-token provider error to the daemon. Other APNs/FCM delivery failures also use fixed provider/status copy, not Apple/Google response bodies. When relay-only Firebase credentials are present, FCM delivery signs an RS256 Google service-account JWT, exchanges it for a cached OAuth token, sends the same hash-only payload contract through HTTP v1 with the same keepalive client policy, and prunes FCM `UNREGISTERED` tokens as stale bindings. Production relay builds do not buffer accepted delivery records after provider dispatch; the in-memory delivered-push list and its metric are compiled only for tests. The production deploy milestone still needs real APNs/FCM credential verification on provider infrastructure.
- The same binary also runs the iroh fallback relay when `FIELDWORK_RELAY_MODE=iroh-relay`. The Ansible scaffold deploys it as `fieldwork-iroh-relay.service` with ACME-backed HTTPS on `:443`, captive-portal HTTP on `:80`, QUIC address discovery on `:7842`, and separate aggregate metrics on `127.0.0.1:9091`.
- Relay push requests are validated with strict Serde schemas and `garde`; unknown fields such as `last_line` are rejected before dispatch, and `session_id_hash`/`session_name_hash` must be lowercase 64-character hex strings. Signed endpoints require `x-fieldwork-signature` over `(method, path, body, nonce, ts_ms)`, reject 5-minute clock skew, and reject nonce reuse.
- Desktop distribution is npm-only. `packages/cli` is the meta package with optional dependencies on `packages/cli-darwin-arm64`, `packages/cli-darwin-x64`, `packages/cli-linux-arm64`, and `packages/cli-linux-x64`.
- The meta package exposes `fieldwork`, `fw`, and `fieldworkd` through npm. `fieldwork` and the short `fw` alias both point at the same CLI dispatcher in `bin/fieldwork`; `fieldworkd` points at `bin/fieldworkd`. During install, `install.js` copies the native `fieldwork` and `fieldworkd` from the matching platform package into the meta package `bin/` directory so the Rust CLI can find the colocated daemon. If postinstall scripts or optional dependencies are disabled, the CLI dispatcher (`fieldwork`/`fw`) and daemon dispatcher fall back to the platform package and print actionable errors. Native help and completion generation use the invoked binary name, so `fw --help` renders `Usage: fw`, `fw completion bash` registers the short alias, and `fieldwork completion bash` registers the long command.
- The Rust CLI has no self-updater. For human-facing commands, it checks the npm registry at most once per day and prints a stderr-only `npm update -g fieldwork` notice when the registry has a newer version. It skips QR, completion, hook, version, and terminal attach flows to keep machine-readable and raw PTY output clean.
- `dist-workspace.toml` keeps cargo-dist in archive/audit mode only: no installers, no Homebrew tap, no updater. GitHub Actions own the npm package assembly and provenance publish path.

The Fieldwork-hosted production relay, APNs/FCM credentials, and release pipeline remain governed by `PLAN.md` and are not fully verified in this local environment.
