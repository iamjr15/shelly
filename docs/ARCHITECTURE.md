# Architecture

The current milestone implements the local host loop:

- `shellyd` listens on a Unix socket at `$XDG_RUNTIME_DIR/shelly/control.sock` or a per-user temp fallback.
- The socket parent is created as `0700`, must be owned by the current user, and symlink parents are rejected.
- The socket file is chmodded to `0600`.
- `shelly` connects over length-prefixed bincode IPC.
- `shellyd` also starts an iroh endpoint for paired mobile clients. It accepts length-prefixed MessagePack frames on a single bidirectional iroh stream and rejects `ClientKind::LocalCli` during `Hello`; desktop CLI traffic is Unix-socket-only.
- The daemon's long-lived iroh secret key is stored in the OS keychain. `SHELLY_IROH_SECRET_KEY_B64` is a deterministic CI/smoke-test override for non-interactive environments that cannot open the keychain; production runs should leave it unset. `SHELLY_IROH_RELAY_URL` can point at a development or Shelly-hosted relay; otherwise the daemon uses the configured iroh default relay map. The Shelly relay deploy scaffold targets AWS Lightsail, while hosted relay DNS and relay-only credentials remain release gates.
- The workspace enables iroh's `platform-verifier` feature, Shelly-owned Reqwest clients use the workspace `rustls` OS-trust path, and relay OTLP uses OpenTelemetry's `reqwest-rustls` native-root feature. `iroh 1.0.0-rc.0` and the relay ACME stack still include `webpki-roots` internally, so security review audits Shelly-owned TLS construction rather than relying on absence of that transitive package.
- The daemon spawns arbitrary commands inside PTYs with `TERM=xterm-256color` and `COLORTERM=truecolor`.
- Each session keeps a 256 KB PTY byte ring and monotonic byte sequence numbers.
- Each live session also maintains a `wezterm-term` terminal model. Warm reconnect replays bytes from the ring; cold or stale attach renders a synthetic ANSI snapshot from the parsed terminal state. The daemon returns the snapshot with the ring `seq` captured under the same ring/terminal lock window, so the next warm replay starts after the bytes represented by the synthetic snapshot rather than duplicating a repaint that arrived during attach.
- Mobile-core tracks `last_seen_seq` from `Attached.seq` and `Output.seq`, both defined as offsets immediately after the bytes carried in that message. Native services cache that offset per session and call `attach_session_from(id, last_seen_seq)` to request warm replay. A daemon-side lag event is terminal for the current output attachment after one `Lag` frame, and the active Android terminal controller reattaches from the latest tracked offset.
- Mobile dashboards use a long-lived `SubscribeSessions` stream that receives complete replacement session lists after local desktop creation/removal and session state changes. Multiple attach clients can subscribe to the same session. Input from any client writes to the PTY, and stale session-scoped input/resize requests receive `Error { NotFound }` instead of being silently accepted.
- Attached client viewports are tracked per connection. The daemon applies the minimum attached rows/columns to the PTY and debounces resize storms at 100 ms; daemon/session tests cover the attach, update, detach, and debounce invariants.
- Session summaries and scrollback are persisted in encrypted `sessions.redb`.
- Paired device records are persisted separately in encrypted `devices.redb` under hashed row keys, so raw device node IDs and push tokens live only inside encrypted row payloads.
- The persisted payload is encrypted with XChaCha20-Poly1305 by default. The encryption key is stored in the OS keychain via `keyring`; if keychain access is unavailable, the daemon does not write plaintext unless the user explicitly opts out with `shelly settings scrollback-encryption off` and restarts `shellyd`.
- The local persistence parent directory is forced to `0700`, the `redb` database file is forced to `0600`, and symlinked persistence directories or database files are rejected before opening either store.
- After daemon restart, dead sessions are restored as last-known sessions. Attaching to one replays encrypted scrollback and then emits `SessionExited`.
- Daemon logs are written to daily `daemon.log*` files under `~/Library/Logs/app.shelly` on macOS or `$XDG_STATE_HOME/shelly` / `~/.local/state/shelly` on Linux. At logging startup, the daemon prunes only `daemon.log*` files older than seven days by modification time.
- Daemon and relay logging both run through a tracing sanitizer layer that drops events marked `privacy.level = "user_content"` before downstream logging layers receive them.
- Daemon and mobile crash reporting are not bundled in v1. `shelly settings telemetry on|off` records local diagnostics consent only, and daemon OTLP/Honeycomb export remains intentionally absent.
- Agent-aware content inference is limited to detected Claude Code and Codex commands. Claude prompt detection keys off explicit approval/permission phrases rather than every question mark; Codex accepts `type`, `event`, and `status` structured event shapes from single JSON objects or JSONL event streams. Unknown command output only uses byte-rate `Idle`/`Working` state.
- The terminal model supplies sanitized last visible line previews for state inference and writes PTY device-status responses back to the child process.
- `SHELLY_SESSION_ID` is injected into spawned PTY environments. Local agent hooks can call `shelly hook claude-stop` or pipe Codex JSON/JSONL into `shelly hook codex-event` to update the owning session over the Unix socket, including shell-backed sessions where the user starts and replaces agents inside one Shelly PTY.
- If `SHELLY_RELAY_CONTROL_URL` is set, the daemon enables relay-mediated push. It stores a long-lived Ed25519 relay-signing key in the OS keychain, registers its public key with the relay, signs push-token registration/unregistration, and sends hashed `AwaitingInput` events when Claude/Codex inference enters `AwaitingInput`. Transport failures and temporary relay failures retry with `backon` exponential backoff for a bounded 60-second budget; signed retries regenerate nonce and timestamp.
- `shelly pair` generates a 5-character Crockford pairing code, builds a compact `PairingTicket { code, node_id, relay_url, addrs, expires_at }`, prints the encoded `sh1…` ticket as a QR plus the bare code for manual entry, shows a live countdown, and holds the desktop approval prompt open. A device pairs by either scanning the QR (which carries reachability and the code) or typing the code (resolved to the same ticket through the relay rendezvous when `SHELLY_RELAY_CONTROL_URL` is configured); both converge on `PairWithCode` over iroh and explicit desktop approval. The daemon owns code generation (`OsRng`), keeps a single active code with a 5-minute TTL, and invalidates that code after 5 wrong in-band attempts. When `SHELLY_RELAY_CONTROL_URL` is set the daemon also best-effort publishes `code → ticket_blob` to the relay; when it is unset the daemon logs a warning and skips publish, so the QR path keeps working and only the typed-code path is unavailable. `CONTRACT_VERSION` is `3`.
- Paired mobile iroh clients may list sessions, attach, send input, resize, detach, and register/unregister push tokens. They cannot create sessions, kill sessions, administer pairing codes, list/remove devices, or emit agent hook events. A client that identifies as `LocalCli` on iroh is rejected before `Welcome`.
- `shelly devices remove <name>` removes the stored device record; that node id receives `Unauthorized` on its next iroh connection.
- `shelly devices remove <name>` also enqueues relay token unregistration when the removed device had a push token.
- `shelly-mobile-core` exposes the iroh client path through UniFFI records, objects, and callback interfaces: `ShellyClient`, `AttachedSession`, `SessionListSink`, and `ByteStreamSink`.
- The mobile-core API intentionally has no create-session or kill-session method.
- `apps/ios` is parked source. It is not part of the current CI, release workflow, or local script surface while iOS implementation is deferred.
- `apps/android` is a Jetpack Compose app target that links the generated UniFFI Kotlin binding and Android `.so` outputs from `cargo-ndk`, stores the paired daemon record encrypted with a non-exportable Android Keystore AES-256-GCM key, excludes pairing and queued FCM-token preference files from backup/transfer, gates resume and stale input through biometric-only BiometricPrompt, renders attached PTY bytes with `connectbot/termlib`, and reattaches from cached `lastSeenSeq` offsets after daemon `Lag` or attached-stream errors.
- The Android app can pair, list/subscribe, attach, send input, resize, detach, and submit/remove FCM tokens through mobile-core. It does not expose session creation, session killing, or command selection.
- Android keeps Firebase Messaging auto-init disabled until pairing, requests `POST_NOTIFICATIONS` and syncs FCM tokens only after a saved or newly approved pairing exists and biometric unlock succeeds, renders foreground FCM messages with fixed generic copy, and routes notification taps through an explicit in-app intent carrying only a `session_id_hash` extra resolved locally after biometric unlock. The app does not export a public deep-link action for session routing. FCM token refresh callbacks only queue trimmed pending tokens in app-private preferences; the paired-and-unlocked ViewModel sync path sends queued/current tokens through mobile-core and clears a queued token only after successful daemon registration. On unpair, the ViewModel best-effort unregisters the pending/current token before clearing local pairing state.
- Android diagnostics sharing is opt-in through Settings or the delayed one-time post-value consent prompt. `MobileTelemetry` persists consent only; no crash-reporting SDK is initialized.
- `shelly-relay` exposes an Axum control plane (in `crates/relay/src/lib.rs`) with `/healthz`, `/v1/version`, `/v1/pair`, `/v1/push/register-token`, `/v1/push/unregister-token`, `/v1/push`, `/v1/pair/publish`, and `/v1/pair/resolve/{code}`. Production deployment serves this listener over Rustls using relay-only `control-plane.crt`/`control-plane.key` systemd credentials; local development can run loopback HTTP when TLS is not required.
- The pairing rendezvous is the relay's optional support for the typed-code path. `POST /v1/pair/publish` is daemon-signed (`x-shelly-signature` over `(method, path, body, nonce, ts_ms)`); it stores `code → ticket_blob` (the opaque encoded `sh1…` ticket) with the daemon-supplied expiry, rejects already-expired publishes, and supersedes any prior code for the same daemon (one active code per daemon). `GET /v1/pair/resolve/{code}` is unauthenticated because the short code is the credential, but it is throttled (20 attempts/min per client), returns a uniform `404` whether the format or the lookup failed, and consumes the code on a correct guess. Throttle buckets key on the connecting socket address; `SHELLY_RELAY_TRUST_FORWARDED_FOR=true` is set only when the relay runs behind a trusted reverse proxy, keying them on the first `x-forwarded-for` hop instead. The daemon still enforces the 5-wrong-attempt cap when the device presents the code over iroh; the relay resolver cannot attribute arbitrary wrong guesses to a specific valid code without creating a denial-of-service vector. Codes are persisted in SQLite alongside the other relay state. The blob is opaque reachability only; no terminal content traverses the relay.
- `shelly-relay` exposes aggregate Prometheus metrics from a separate metrics app/listener, defaulting to `127.0.0.1:9090`; the listener can be disabled locally with `SHELLY_RELAY_METRICS_ADDR=off`.
- `shelly-relay` can export OTLP HTTP/protobuf traces from the relay host. Honeycomb is configured with `SHELLY_RELAY_OTLP_ENDPOINT`, `SHELLY_RELAY_OTLP_SAMPLE_RATE` (default `0.01`), optional `SHELLY_RELAY_HONEYCOMB_DATASET`, and a relay-only `honeycomb-api-key` systemd credential or `SHELLY_RELAY_HONEYCOMB_API_KEY_PATH`. The relay records only static endpoint names, service metadata, and fixed enum fields such as push platform and event type.
- The relay persists daemon public keys, push-token ownership, and recent replay nonces in SQLite at `SHELLY_RELAY_DB_PATH` (default `/var/lib/shelly/relay.db`). Push-token ownership rows carry a last-used timestamp, are refreshed on accepted push dispatch, and are pruned after 90 days with no use. The DB directory is chmodded `0700`, and the main DB plus SQLite `-wal`/`-shm` sidecars are chmodded `0600` on Unix.
- Per-daemon push rate counters use a bounded `moka` TTL cache. When relay-only Firebase credentials are present, FCM delivery signs an RS256 Google service-account JWT, exchanges it for a cached OAuth token, sends the hash-only payload contract through HTTP v1 with keepalive client policy, and prunes FCM `UNREGISTERED` tokens as stale bindings. Provider failures use fixed provider/status copy, not Google response bodies. Production relay builds do not buffer accepted delivery records after provider dispatch; the in-memory delivered-push list and its metric are compiled only for tests. APNs support remains relay-only implementation for the deferred iOS path, not a current local release gate.
- The same binary also runs the iroh fallback relay when `SHELLY_RELAY_MODE=iroh-relay`. The Ansible scaffold deploys it as `shelly-iroh-relay.service` with ACME-backed HTTPS on `:443`, captive-portal HTTP on `:80`, QUIC address discovery on `:7842`, and separate aggregate metrics on `127.0.0.1:9091`.
- Relay push requests are validated with strict Serde schemas and `garde`; unknown fields such as `last_line` are rejected before dispatch, and `session_id_hash`/`session_name_hash` must be lowercase 64-character hex strings. Signed endpoints require `x-shelly-signature` over `(method, path, body, nonce, ts_ms)`, reject 5-minute clock skew, and reject nonce reuse.
- Desktop distribution is npm-only. `packages/cli` is the `shellykit` meta package with optional dependencies on `packages/cli-darwin-arm64`, `packages/cli-darwin-x64`, `packages/cli-linux-arm64`, and `packages/cli-linux-x64`.
- The meta package exposes `shelly` and `shellyd` through npm. `shelly` points at the CLI dispatcher in `bin/shelly`; `shellyd` points at `bin/shellyd`. During install, `install.js` copies the native `shelly` and `shellyd` from the matching platform package into the meta package `bin/` directory so the Rust CLI can find the colocated daemon. If postinstall scripts or optional dependencies are disabled, the CLI and daemon dispatchers fall back to the platform package and print actionable errors. Native help and completion generation use `shelly`.
- The Rust CLI has no self-updater. For human-facing commands, it checks the npm registry at most once per day and prints a stderr-only `npm update -g shellykit` notice when the registry has a newer version. It skips QR, completion, hook, version, and terminal attach flows to keep machine-readable and raw PTY output clean.
- `dist-workspace.toml` keeps cargo-dist in archive/audit mode only: no installers, no Homebrew tap, no updater. GitHub Actions own the npm package assembly and provenance publish path.

The Shelly-hosted production relay, FCM credentials, and release pipeline remain governed by `PLAN.md` and are not fully verified in this local environment.
