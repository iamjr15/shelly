# Privacy

Current implemented behavior includes local IPC plus iroh pairing/listing for paired remote clients.

Store-submission privacy labels are tracked separately in `docs/STORE_PRIVACY.md`. That file is an answer sheet for App Store Connect and Google Play Data safety; the console submissions remain release gates until real signed mobile builds and account access are available.

- Terminal input and output stay on the host unless a paired remote iroh client attaches to a session.
- Over iroh, terminal/session protocol frames are carried inside encrypted QUIC connections. The iroh relay path can see endpoint IPs, timing, and byte counts, but not PTY content, keystrokes, commands, paths, or session metadata.
- The QR payload contains the daemon node id, relay URL, direct addresses, pair token, and expiry. Pair tokens are single-use and expire after 10 minutes.
- No daemon telemetry is exported by default. The daemon initializes Sentry crash reporting only when telemetry is enabled through `fieldwork settings telemetry on --sentry-dsn <dsn>` or the `FIELDWORK_TELEMETRY_OPT_IN=true` plus `FIELDWORK_SENTRY_DSN` environment override. Sentry is configured with `send_default_pii=false` and `traces_sample_rate=0.0`; daemon OTLP trace export is intentionally absent from v1.
- The CLI update notice contacts the public npm registry at most once per day to read `fieldwork/latest`, caches only the checked timestamp and latest version in the private Fieldwork config directory, and never downloads code. Set `FIELDWORK_DISABLE_UPDATE_CHECK=1` to disable the check.
- The daemon and relay install a tracing privacy sanitizer layer. Any event explicitly marked `privacy.level = "user_content"` is dropped before it reaches daemon file logs or relay stderr logs.
- The iOS app stores the paired daemon node id, relay/direct addresses, device node id, and device secret key in the data-protection iOS Keychain with `WhenUnlockedThisDeviceOnly` accessibility.
- The iOS app requires biometric-only LocalAuthentication on app resume after 5 minutes and before stale terminal input is sent. Its locked root does not render session or terminal views, and paired session/push activation waits for successful unlock.
- The iOS app starts Sentry crash reporting only after explicit opt-in through Settings or the one-time delayed consent prompt shown after an `AwaitingInput` response produces at least 10 output lines, and only when the release build contains a `FieldworkSentryDsn` value injected from release secrets. It sets `sendDefaultPii=false`, `tracesSampleRate=0.0`, and closes the SDK again when the toggle is turned off.
- The Android app stores the paired daemon node id, relay/direct addresses, device node id, and device secret key in Android Keystore-backed `EncryptedSharedPreferences`; the pairing preferences file and queued FCM-token preferences file are excluded from full backup, cloud backup, and device transfer.
- The Android app requires biometric-only BiometricPrompt on app resume after 5 minutes and before stale terminal input is sent. Its locked root does not render session or terminal views, and paired session/push activation waits for successful unlock.
- The Android app starts Sentry crash reporting only after explicit opt-in through Settings or the one-time delayed consent prompt shown after an `AwaitingInput` response produces at least 10 output lines, and only when the release build contains `BuildConfig.FIELDWORK_SENTRY_DSN` injected from release secrets. It disables Sentry auto-init in the manifest, sets `sendDefaultPii=false`, `tracesSampleRate=0.0`, and closes the SDK again when the toggle is turned off.
- APNs/FCM token registration plumbing exists in the mobile apps and mobile-core. Paired phones send tokens only over the encrypted iroh channel to the daemon after biometric unlock. The Android app disables Firebase Messaging auto-init in the manifest; it enables token generation only after the app has a paired daemon, biometric unlock has succeeded, and a Firebase `google-services.json` configuration is present. Android FCM token refresh callbacks queue trimmed tokens in app-private `fieldwork_push_tokens.xml`, which is excluded from backup and transfer, and the queued token is sent and cleared only by the paired-and-unlocked token sync path.
- Native notification UI uses fixed generic copy. Notification tap routing carries only lowercase 64-character hex `session_id_hash`; the apps reject malformed hashes and resolve accepted hashes against locally fetched sessions after biometric unlock.
- Scrollback is persisted locally in `redb` and encrypted with XChaCha20-Poly1305.
- Paired device records and registered push tokens are persisted locally in a separate encrypted `devices.redb` under hashed row keys, so raw device node IDs and push tokens live only inside encrypted row payloads.
- Local persistence directories are forced to `0700`, `redb` database files are forced to `0600`, and symlinked persistence directories or database files are rejected before opening.
- The persistence key is stored in the OS keychain. If keychain access is unavailable, the daemon refuses to write plaintext persistence and runs without persisted scrollback unless the user explicitly runs `fieldwork settings scrollback-encryption off` and restarts `fieldworkd`.
- The daemon iroh secret key is stored in the OS keychain. If it cannot be loaded or created, iroh pairing is unavailable while the local Unix-socket workflow continues. `FIELDWORK_IROH_SECRET_KEY_B64` exists for deterministic CI/smoke tests in non-interactive environments; production deployments should not set it.
- The daemon relay-signing key is stored in the OS keychain. If `FIELDWORK_RELAY_CONTROL_URL` is not set, relay push dispatch is disabled and registered push tokens remain local only.
- When relay push is enabled, the daemon sends push tokens to the relay for ownership binding and sends `AwaitingInput` push requests containing only `recipient_token`, `platform`, `session_id_hash`, `session_name_hash`, fixed `event_type`, `nonce`, and `ts_ms`. Removing a paired device also sends a signed relay token-unregistration request if that device had a saved push token. The session hashes are validated by the relay as lowercase 64-character hex strings. It does not send terminal bytes, command lines, paths, session names, or `last_line`.
- The relay persists daemon public keys, push-token ownership, and recent replay nonces in SQLite at `FIELDWORK_RELAY_DB_PATH` (default `/var/lib/fieldwork/relay.db`). On Unix the DB directory is `0700`, and the main DB plus SQLite `-wal`/`-shm` sidecars are `0600`.
- The local relay rejects unknown push fields through strict schema validation, tests rejection of `last_line`, `command`, `path`, and `session_name` fields, and rejects non-hex or uppercase session hashes. It also verifies daemon signatures, token ownership, nonce replay, timestamp skew, and per-daemon rate limits.
- Production relay builds do not retain accepted provider delivery records after dispatch. The in-memory delivered-push list used by local tests is `#[cfg(test)]`, so production state contains only the daemon key, token ownership, replay-window, rate-limit, and aggregate metric state required to operate the service.
- Daemon-facing relay API errors for APNs/FCM delivery use fixed provider/status copy and do not reflect Apple/Google response bodies. Local tests inject a provider body containing path and `last_line` sentinels and verify those strings are not returned.
- The relay exposes aggregate Prometheus metrics on its metrics listener. Metrics include only counters and gauges such as accepted daemon registrations, active token count, and accepted privacy-preserving push requests. Metrics do not include daemon node IDs, push tokens, session hashes, commands, paths, terminal bytes, IP addresses, or user-facing names.
- The relay can export 1% sampled OTLP traces to Honeycomb from the relay host when `FIELDWORK_RELAY_OTLP_ENDPOINT` and a relay-only `honeycomb-api-key` systemd credential are configured. Relay span fields are aggregate/static only: endpoint name, platform enum, event type enum, service name, and service version. They do not include daemon node IDs, push tokens, session hashes, commands, paths, terminal bytes, IP addresses, or user-facing names. The Honeycomb API key is read from `LoadCredential=honeycomb-api-key` or `FIELDWORK_RELAY_HONEYCOMB_API_KEY_PATH`, and the config debug output redacts header values.
- APNs `.p8` and FCM service-account JSON are not present in this repository or daemon/mobile code. Provider signing is implemented only inside the relay process: APNs reads the relay-only `.p8` and caches an ES256 provider JWT for 50 minutes; FCM reads the relay-only Firebase service-account JSON, signs an RS256 service-account JWT, and caches the resulting OAuth token. Both provider payloads use fixed alert copy plus opaque session hashes. Real provider verification remains blocked in this shell by missing relay-only Apple/Firebase credentials and physical-device testing.

Push notification payload privacy rules from `PLAN.md` remain binding: push payloads must contain only opaque hashes and fixed enum-derived text, never terminal content, command lines, paths, session names, or `last_line`.
